---

The Verifications API is the primary entry point for all credential verification activity on TrustOS. You create a verification event by referencing a policy, choose a delivery channel for the holder interaction, and then either poll for the result or receive it asynchronously via webhook. The response never exposes raw credential claim values — only structured outcomes, scores, and an optional compute output.

All requests must be authenticated with mTLS and an OAuth 2.0 client-credentials bearer token scoped to your verifier DID. Every `POST` endpoint accepts an optional `Idempotency-Key` header; replays within the 24-hour window return the original response without creating a duplicate event.

---

## POST /verifications

Start a new verification event. TrustOS creates an OID4VP presentation request, generates the channel-appropriate delivery artifact (QR code, deep link, or USSD session), and returns immediately with a `requestId` you can use to poll for the result.

### Request body

<ParamField body="policyId" type="string" required>
  The ID of the policy that governs this verification. The policy must belong to your verifier. The policy version active at request creation is pinned for the lifetime of the event.
</ParamField>

<ParamField body="holderRef" type="string" required>
  A pseudonymous, stable reference to the holder whose credentials should be verified. TrustOS resolves this to the holder's registered DID before presentation proof verification. Use the identifier your system assigned to the individual at enrolment — never a raw PII value.
</ParamField>

<ParamField body="channel" type="string">
  Delivery channel for the holder interaction. One of: `direct_post`, `qr`, `cross_device`, `ussd`. Defaults to `direct_post` if omitted.
</ParamField>

<ParamField body="sharingTemplateId" type="string">
  Optional ID of a sharing template that overrides the default template defined on the policy. The template controls which derived fields are returned to you and how claim inputs are combined.
</ParamField>

<ParamField body="computePayload" type="object">
  Optional inline compute request. When provided, TrustOS executes the referenced artifact inside a confidential compute sandbox against the resolved credential claims and includes the output in the result. The artifact must be registered as a compute manifest before it can be referenced.

  <Expandable title="computePayload fields">
    <ParamField body="artifactRef" type="string" required>
      The `manifestId` of the registered compute manifest to execute.
    </ParamField>
    <ParamField body="kind" type="string" required>
      The type of compute artifact. One of: `wasm`, `script`, `ml_model`.
    </ParamField>
    <ParamField body="outputType" type="string" required>
      The expected shape of the compute output. One of: `numeric`, `boolean`, `bucket`, `json`.
    </ParamField>
  </Expandable>
</ParamField>

### Example request

```bash
curl -X POST https://api.trustos.example/v1/verifications \
  --cert client.crt --key client.key \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: a8098c1a-f86e-11da-bd1a-00112444be1e" \
  -d '{
    "policyId": "pol_01JAFFORDABILITY",
    "holderRef": "holder_7f2a9c",
    "channel": "qr",
    "computePayload": {
      "artifactRef": "mft_lender_affordability",
      "kind": "wasm",
      "outputType": "bucket"
    }
  }'
```

### Response — 201 Created

```json
{
  "requestId": "req_01J...",
  "status": "pending",
  "presentationRequest": {
    "requestId": "req_01J...",
    "nonce": "n_8f2...",
    "verifierId": "did:web:lender.example",
    "requiredCredentials": ["Identity", "Employment", "BankKey"],
    "responseMode": "direct_post",
    "createdAt": "2026-06-24T10:00:00Z",
    "expiresAtMs": 1750766700000
  },
  "channelEnvelope": {
    "channel": "qr",
    "requestUri": "https://api.trustos.example/oid4vp/req_01J.../request",
    "qr": "data:image/png;base64,...",
    "expiresAt": "2026-06-24T10:05:00Z"
  }
}
```

<ResponseField name="requestId" type="string">
  Unique identifier for this verification event. Use it with `GET /verifications/{id}` to poll for the result or to correlate webhook deliveries.
</ResponseField>

<ResponseField name="status" type="string">
  Initial status of the event. Always `"pending"` on creation.
</ResponseField>

<ResponseField name="presentationRequest" type="object">
  The OID4VP presentation request object that governs the holder interaction.

  <Expandable title="presentationRequest fields">
    <ResponseField name="requestId" type="string">
      Echoes the top-level `requestId`.
    </ResponseField>
    <ResponseField name="nonce" type="string">
      Cryptographic nonce bound to this presentation request. The holder's wallet includes this in the presentation proof.
    </ResponseField>
    <ResponseField name="verifierId" type="string">
      The DID of your verifier, as registered in the Trust Registry.
    </ResponseField>
    <ResponseField name="requiredCredentials" type="array">
      The credential types the holder must present, derived from the governing policy.
    </ResponseField>
    <ResponseField name="responseMode" type="string">
      The OID4VP response mode. Always `"direct_post"`.
    </ResponseField>
    <ResponseField name="createdAt" type="string">
      ISO-8601 timestamp of when the presentation request was created.
    </ResponseField>
    <ResponseField name="expiresAtMs" type="integer">
      Unix epoch milliseconds at which the presentation request expires.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="channelEnvelope" type="object">
  Channel-specific delivery artifact the holder uses to respond to the verification request.

  <Expandable title="channelEnvelope fields">
    <ResponseField name="channel" type="string">
      The channel this event was created on: `direct_post`, `qr`, `cross_device`, or `ussd`.
    </ResponseField>
    <ResponseField name="requestUri" type="string">
      The OID4VP request URI the holder's wallet fetches to retrieve the presentation definition.
    </ResponseField>
    <ResponseField name="qr" type="string">
      Base64-encoded PNG QR code image. Present on `qr` and `cross_device` channels only.
    </ResponseField>
    <ResponseField name="expiresAt" type="string">
      ISO-8601 timestamp after which the presentation request is no longer valid.
    </ResponseField>
  </Expandable>
</ResponseField>

## GET /verifications/{id}

Poll for the current status or final result of a verification event. Call this endpoint after creating a verification if you are not using webhook delivery.

### Path parameters

<ParamField path="id" type="string" required>
  The `requestId` returned by `POST /verifications`.
</ParamField>

### Example request

```bash
curl https://api.trustos.example/v1/verifications/req_01J... \
  --cert client.crt --key client.key \
  -H "Authorization: Bearer $ACCESS_TOKEN"
```

### Response — 200 OK (pending)

```json
{
  "requestId": "req_01J...",
  "status": "pending",
  "policyId": "pol_01JAFFORDABILITY",
  "channel": "qr",
  "expiresAtMs": 1750766700000
}
```

### Response — 200 OK (completed)

```json
{
  "status": "completed",
  "requestId": "req_01J...",
  "policyId": "pol_01JAFFORDABILITY",
  "policyVersion": 3,
  "decision": "Approve",
  "riskScore": 0.12,
  "trustScore": 88,
  "riskFlags": [],
  "proxy": "px_9f3...",
  "computeOut": { "affordabilityBand": "high" },
  "reasons": ["all required credentials valid and policy met"]
}
```

<ResponseField name="requestId" type="string">
  The unique identifier for this verification event, echoed from the creation response.
</ResponseField>

<ResponseField name="status" type="string">
  Current lifecycle status of the event. See the status table below.
</ResponseField>

<ResponseField name="decision" type="string">
  Policy decision when `status` is `"completed"`. One of: `Approve`, `Decline`, `Review`.
</ResponseField>

<ResponseField name="policyId" type="string">
  The ID of the policy that governed this verification.
</ResponseField>

<ResponseField name="policyVersion" type="integer">
  The version of the policy that was pinned when the verification was created.
</ResponseField>

<ResponseField name="riskScore" type="number">
  Fraud and risk score between `0.0` (lowest risk) and `1.0` (highest risk). Present when `status` is `"completed"`.
</ResponseField>

<ResponseField name="trustScore" type="number">
  Composite trust score between `0` and `100`. Present when `status` is `"completed"`.
</ResponseField>

<ResponseField name="riskFlags" type="array">
  List of string risk flags raised during evaluation, e.g. `"fraudScore>0.6"`. Empty when none are raised.
</ResponseField>

<ResponseField name="proxy" type="string">
  A pseudonymous, stable identifier for the holder. Use this to correlate repeat interactions without storing PII.
</ResponseField>

<ResponseField name="computeOut" type="object | null">
  The output of the confidential compute execution, if a `computePayload` was supplied. `null` when no compute was requested. Shape is defined by the registered manifest's `outputSchema`.
</ResponseField>

<ResponseField name="reasons" type="array">
  Human-readable strings explaining the decision.
</ResponseField>

<ResponseField name="expiresAtMs" type="integer">
  Unix epoch milliseconds at which the pending verification expires. Present only when `status` is `"pending"`.
</ResponseField>

<ResponseField name="channel" type="string">
  The delivery channel for this verification. Present when `status` is `"pending"`.
</ResponseField>

### Status values

| Status | Description |
|---|---|
| `pending` | The OID4VP request has been issued; waiting for the holder to present credentials. |
| `completed` | The holder presented credentials and a decision was reached. Check `decision`. |
| `expired` | The presentation window closed before the holder responded. |
| `error` | An unexpected error occurred during processing. Retry or contact support with the `traceId`. |

<Note>
  The result never contains raw credential claim values. TrustOS returns only derived outcomes, scores, and compute outputs — your integration code should never expect or store raw claims.
</Note>

## Error responses

All errors follow [RFC 9457](https://www.rfc-editor.org/rfc/rfc9457) `application/problem+json`. Every problem response includes a `traceId` you can share with support.

| HTTP | `type` | When |
|---|---|---|
| 400 | `invalid-request` | Malformed body or schema violation |
| 401 | `unauthenticated` | Missing or invalid mTLS / bearer token |
| 403 | `verifier-not-approved` | Verifier is not approved in the Trust Registry |
| 404 | `not-found` | Unknown `requestId` or `policyId` |
| 409 | `conflict` | `Idempotency-Key` replay with a different request body |
| 410 | `presentation-expired` | OID4VP session expired before the holder responded |
| 422 | `policy-unsatisfiable` | Policy references an unknown credential type or manifest |
| 429 | `rate-limited` | Per-verifier quota exceeded. Respect the `Retry-After` header. |
| 502/504 | `dependency-failure` | Upstream dependency unavailable and cache is stale |
