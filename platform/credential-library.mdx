TrustOS recognises 17 credential types out of the box, covering identity, employment, financial standing, licensing, and more. Every type is a W3C Verifiable Credential conforming to the VCDM 2.0 standard — no proprietary formats. All credential schemas and their JSON-LD contexts are published to the Marketplace, meaning new types can be added without any code changes or platform deployments. If your use case requires a credential type not listed here, the Custom/Scheme type lets any approved issuer publish and register their own schema.

## Common fields on every credential

Regardless of type, every credential issued through TrustOS includes these five fields on the credential subject. They carry platform-wide semantics that drive policy evaluation, fraud detection, and privacy-preserving repeat interactions.

| Field | Type | Meaning |
|---|---|---|
| `assuranceLevel` | integer 0–4 | Strength of the identity proofing behind this credential. Your policy sets a minimum; presentations below it are declined. |
| `proxy` | string (opaque) | A pseudonymous, stable identifier your system can use to recognise a returning subject without storing any PII. |
| `safetyIndicator` | object `{signal, source, timestamp}` | An issuer-detected risk flag carried on the credential. Feeds into the Decision Engine's risk flags. Nullable. |
| `validFrom` | datetime | The earliest moment the credential is valid. Presentations before this time are declined at pipeline step 2. |
| `validUntil` | datetime | The expiry time. Presentations after this time are declined at pipeline step 2. |

## Assurance level scale

The `assuranceLevel` value tells you how strongly the issuing authority verified the subject's identity before issuing the credential. Your policy's `minAssurance` setting filters out credentials that don't meet your bar.

| Level | Meaning |
|---|---|
| 0 | Self-asserted — the subject provided the information without any verification |
| 1 | Single weak factor verified (e.g. email address or phone number confirmed) |
| 2 | Document-verified — an identity document was checked, but no biometric liveness |
| 3 | Document + biometric liveness — both the document and the subject's live presence were verified |
| 4 | In-person or supervised proofing by an accredited authority |

## Credential catalogue

<Accordion title="Identity">
  **Typical issuer:** Government ID authority

  | Claim | Description |
  |---|---|
  | `fullName` | Legal full name of the credential subject |
  | `idNumber` | National identity document number |
  | `dateOfBirth` | Subject's date of birth |
  | `photoHash` | A cryptographic hash of the subject's biometric photograph — the platform never receives or stores the image itself |
  | `livenessVerified` | Boolean attestation by the issuer that a biometric liveness check was passed |
  | `assuranceLevel` | Typically 3 or 4 for government-issued identity credentials |
</Accordion>

<Accordion title="Custom / Scheme Credential">
  **Typical issuer:** Any approved issuer

  Custom credential types allow any approved issuer to define their own credential schema and publish it to the Marketplace. The schema follows the same W3C VCDM 2.0 envelope and must declare which claims are independently disclosable. Once published, the type becomes available for verifiers to reference in policies — no platform deployment required.

  Use this type for industry-specific, scheme-specific, or domain-specific credentials that are not covered by the 16 standard types above.
</Accordion>

<Tip>
  You can publish a Custom credential type to the Marketplace directly from the TrustOS console. Once approved, other verifiers on the platform can reference it in their own policies — making it reusable across your ecosystem without any code changes.
</Tip>

## A note on biometric data

The `photoHash` field on the Identity credential is a cryptographic hash of the holder's biometric photograph — not the image itself. TrustOS never receives, processes, or stores biometric images at any point in the verification flow. The `livenessVerified` boolean is an attestation by the issuing authority that a liveness check occurred; the underlying biometric data remains with the issuer.
